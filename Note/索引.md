# 一、基础概念
### 核心前提
- **数据文件**：底层数据按关键字**有序排列**（稠密/稀疏索引都依赖数据有序）
- **索引项结构**：`(索引关键字, 数据物理地址/指针)`
- 存储底座：**数组存索引目录（随机快查）+ 链表串数据/块内元素（动态增删）**

---

# 二、稠密索引 (Dense Index)
## 1. 定义
**每一条数据记录，对应一条索引项**；索引个数 = 数据记录条数。
> 数组：完整存放所有索引项（有序）
> 链表：串联全局所有有序原始数据

## 2. 结构图解
```
【索引数组(稠密，一一对应)】
[K1,ptr1] → [K2,ptr2] → [K3,ptr3] → [K4,ptr4]

↓ptr1      ↓ptr2      ↓ptr3      ↓ptr4
【全局有序数据链表】
(Data1) → (Data2) → (Data3) → (Data4)
```

## 3. 工作原理
1. 数据链表：所有记录按 Key 从小到大**全局有序链式存储**
2. 稠密索引数组：为**每一个数据节点**单独建一条索引，数组按 Key 有序
3. 查询流程：
   - 对索引数组**二分查找 O(logN)** → 拿到数据指针
   - 直接定位链表目标数据节点

## 4. 优缺点
✅ 优点：
- 查询快：一次二分索引就定位，不用遍历整块
- 等值/范围查询都友好
❌ 缺点：
- 索引体积大：数据越多，索引数组越长，内存开销高
- 插入/删除数据：不仅改链表，还要**插入/挪动索引数组元素**，开销大

## 5. 典型场景
小体量有序数据集、简单内存索引表。

---

# 三、稀疏索引 (Sparse Index)
## 1. 定义
把有序数据**分块存储**，**每一块只建一条索引项**；索引个数 ≪ 数据记录数。
> 数组：存「块首关键字+块指针」的分级索引目录
> 链表：每个块内部用链表串多条数据

## 2. 结构图解
```
【稀疏索引数组(只存块索引)】
[块1首K1, 块1指针]  [块2首K5, 块2指针]  [块3首K9, 块3指针]

↓块1指针
块1链表：(K1,D1)→(K2,D2)→(K3,D3)→(K4,D4)
↓块2指针
块2链表：(K5,D5)→(K6,D6)→(K7,D7)→(K8,D8)
↓块3指针
块3链表：(K9,D9)→……
```

### 关键规则：
- 块内数据 Key 都在 `[块首Key, 下一块首Key)` 区间
- 索引数组只存**块边界**，不存每条数据

## 3. 查询流程
1. 二分**稀疏索引数组** → 确定目标 Key 属于哪一个数据块 O(logM)，M≪N
2. 根据块指针，进入对应**块内链表**遍历查找 O(块内平均长度)

## 4. 优缺点
✅ 优点：
- 索引极小：大大节省内存/磁盘空间
- 数据增删只改**块内链表**，一般不用改动上层索引数组
❌ 缺点：
- 定位到块后需要遍历块内链表，速度比稠密索引慢
- 范围查询要跨块处理

## 5. 典型场景
大数据量、磁盘文件索引、数据库多级非聚簇索引外层、日志有序分块存储。

# 四、稠密索引 vs 稀疏索引 核心对比
| 对比维度 | 稠密索引 | 稀疏索引 |
| :--- | :--- | :--- |
| 索引项数量 | 与数据记录**1:1** | 按块建立，索引少很多 |
| 索引数组大小 | 大 | 小，节省空间 |
| 查询速度 | 快（仅二分索引） | 稍慢（定块+块内遍历） |
| 插入删除开销 | 大（需维护索引数组有序） | 小（仅改块内链表） |
| 依赖数据 | 全局严格有序 | 全局有序+按块划分 |
| 适用规模 | 小数据量 | 海量磁盘/文件大数据 |

---

# 五、结合「数组+链表」的底层分工总结
1. **稠密索引**
   - 数组：存**全量精细索引**，二分快速精准定位
   - 链表：串联全部有序原始数据，负责动态增删
2. **稀疏索引**
   - 数组：存**分块粗粒度目录**，用少量索引覆盖大量数据
   - 链表：每个块内部链式存多条数据，平衡存储与动态修改

---

# 补充拓展：多级稀疏索引
当一级稀疏索引数组还是太大，可以再对**一级索引数组建二级稀疏索引**，形成多级索引（数据库、文件系统经典架构），本质还是：
`高层稀疏索引数组 → 下层块/索引 → 最终链表数据`
